# Threads

A thread is a persistent conversation container that holds an ordered sequence of messages between a user and an agent. Threads provide the memory layer for AI interactions — they store context, enable conversation history retrieval, and support branching to explore alternative conversation paths.

## Key Properties

| Property | Description |
|----------|-------------|
| `id` | Unique identifier |
| `agentId` | The agent this thread is associated with |
| `orgId` | Owning organization |
| `metadata` | Arbitrary key-value data attached to the thread |
| `status` | Current state: active, archived |
| `createdAt` | Timestamp of creation |
| `updatedAt` | Timestamp of last activity |

## Thread Lifecycle

Threads follow a natural progression:

1. **Create** — Initialize a thread linked to a specific agent. The thread starts empty with no messages.
2. **Messages** — Add user messages and receive agent responses. Each message is persisted immediately and the full history is available at any time.
3. **Branch** — Fork the conversation from any message to explore a different direction without losing the original path.
4. **Archive** — Mark a thread as archived when the conversation is complete. Archived threads remain readable but no longer accept new messages.

## Message Structure

Each message in a thread contains:

| Field | Description |
|-------|-------------|
| `id` | Unique message identifier |
| `threadId` | Parent thread |
| `role` | Message author: `user`, `assistant`, `system`, or `tool` |
| `content` | The message text or structured content |
| `metadata` | Optional key-value data (e.g., token counts, model used) |
| `createdAt` | Timestamp |

Messages are immutable once created. The conversation history is an append-only log, ensuring a reliable audit trail of every interaction.

### Message Roles

- **user** — Input from the human or calling application
- **assistant** — Response generated by the agent's LLM
- **system** — System-level instructions (typically the agent's system prompt, injected automatically)
- **tool** — Results returned from tool/function execution during agent processing

## Thread Branching

Branching lets you fork a conversation from any point in its history. This is useful for:

- **Exploring alternatives** — Try a different prompt or approach without losing the original conversation
- **A/B testing** — Compare how an agent responds to variations of the same question
- **Error recovery** — Branch from before a problematic response and retry with adjusted context

When you branch a thread, a new thread is created containing all messages up to and including the branch point. The original thread is unchanged. Both threads can continue independently from that point.

```
Original Thread:
  msg1 -> msg2 -> msg3 -> msg4 -> msg5

Branch from msg3:
  Original: msg1 -> msg2 -> msg3 -> msg4 -> msg5
  Branch:   msg1 -> msg2 -> msg3 -> (new messages...)
```

## API Examples

### Create a Thread

```bash
curl -s -X POST \
  -H "Authorization: Bearer tdsk_your_key" \
  -H "Content-Type: application/json" \
  -d '{
    "agentId": "agent_id_here"
  }' \
  https://api.threadedstack.app/_/threads
```

### Add a Message

```bash
curl -s -X POST \
  -H "Authorization: Bearer tdsk_your_key" \
  -H "Content-Type: application/json" \
  -d '{
    "role": "user",
    "content": "Explain how thread branching works."
  }' \
  https://api.threadedstack.app/_/threads/thread_id/messages
```

### List Messages in a Thread

```bash
curl -s -H "Authorization: Bearer tdsk_your_key" \
  https://api.threadedstack.app/_/threads/thread_id/messages
```

### Branch a Thread

```bash
curl -s -X POST \
  -H "Authorization: Bearer tdsk_your_key" \
  -H "Content-Type: application/json" \
  -d '{
    "messageId": "branch_from_message_id"
  }' \
  https://api.threadedstack.app/_/threads/thread_id/branch
```

### List Threads for an Agent

```bash
curl -s -H "Authorization: Bearer tdsk_your_key" \
  "https://api.threadedstack.app/_/threads?agentId=agent_id_here"
```
