# Threads

## What are Threads

Threads are persistent conversation containers in Threaded Stack. Each thread holds an ordered sequence of messages that represent a single conversational session.

Threads are scoped to an **organization**. They can optionally be associated with a **project** and a **provider** (the LLM backend used for generation).

Key characteristics:

- **User-owned**: Every thread has a `userId` owner. Only the owner can read, update, or delete their threads and messages.
- **Organization-scoped**: Threads belong to an org (`orgId`), which controls quota tracking and permission checks.
- **Branchable**: Threads support parent/child relationships for conversation branching (see [Thread Branching](#thread-branching)).
- **Quota-tracked**: Creating threads and messages increments the org's billing-period usage counters for `threads` and `messages` resource types.

## Message Types

### Message Type Enum

| Type | Value | Description |
|------|-------|-------------|
| `user` | `"user"` | Messages sent by the human user |
| `assistant` | `"assistant"` | Responses generated by the AI model |
| `system` | `"system"` | System-level instructions or context injected into the conversation |
| `tool` | `"tool"` | Messages representing tool invocations or their results |
| `action` | `"action"` | Messages representing discrete actions taken by the AI |

### Content Format

Message content is stored as a structured array of typed content blocks. Each block has a `type` discriminator. A single message can contain multiple content blocks of different types.

**Content block types:**

| Type | Key Fields | Description |
|------|------------|-------------|
| `text` | `text: string` | Plain text or markdown content |
| `image` | `data: string`, `mimeType: string` | Base64-encoded image data |
| `file` | `assetId: string`, `fileName: string` | Reference to an uploaded file asset |
| `tool_use` | `id: string`, `name: string`, `input: Record<string, unknown>` | A tool invocation request from the assistant |
| `tool_result` | `content: string`, `toolUseId: string`, `isError?: boolean` | The result of a tool invocation |
| `artifact` | `content: string`, `title?: string`, `language?: string`, `artifactType: TArtifactType` | A generated artifact (code, HTML, SVG, markdown, etc.) |
| `thinking` | `thinking: string`, `thinkingSignature?: string`, `redacted?: boolean` | Internal reasoning/chain-of-thought from the model |

## Thread Branching

Thread branching allows users to fork a conversation at any message point, creating a new thread that inherits the history up to (and including) the branch point.

### How It Works

1. The user specifies a thread and a message where they want to branch.
2. The system creates a new thread that copies all messages from the start of the conversation up to and including the selected message.
3. The new thread links back to the original via a parent reference and records which message it branched from.
4. The original thread remains unchanged.

### API Support

The `GET /:id` endpoint supports an `?include=` query parameter:
- `?include=branches` -- returns child threads as a `branches` array on the response
- `?include=parent` -- returns the parent thread as a `parentThread` object (with access control: same org, same user)
- `?include=branches,parent` -- both

## Backend API

Thread endpoints are scoped under organizations:

```http
/_/orgs/:orgId/threads
```

Authentication is required for all endpoints (JWT or API key). Permission checks use the org-level RBAC system. The thread owner (`userId`) is enforced on all read/write operations -- users can only access their own threads.

### Thread Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/` | Create a new thread. Increments the org's `threads` quota counter. |
| `GET` | `/` | List threads scoped to the authenticated user. Supports `?limit` and `?offset` pagination. |
| `GET` | `/:id` | Get a single thread by ID. Supports `?include=branches,parent` for related data. |
| `PUT` | `/:id` | Update thread `name`, `meta`, or `public` flag. |
| `DELETE` | `/:id` | Delete a thread and all its messages (cascade). |
| `POST` | `/:threadId/branch` | Branch a thread at a specific message. Body: `{ messageId: string }`. |

### Message Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/:threadId/messages` | Create a message in a thread. Body: `{ type, content }`. Increments the org's `messages` quota counter. |
| `GET` | `/:threadId/messages` | List messages for a thread. Supports `?limit` and `?offset` pagination. |
| `PUT` | `/:threadId/messages/:messageId` | Update a message's `content`, `type`, or `meta`. |
| `DELETE` | `/:threadId/messages/:messageId` | Delete a specific message from a thread. |

### File Endpoint

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/:threadId/files` | Upload a file to a thread. Body: `{ fileName, data (base64), mimeType }`. Creates an asset record, extracts text content, and returns extracted text and asset metadata. Max file size: 25 MB. |

## Authorization Model

Every thread endpoint follows the same authorization pattern:

1. **Authentication**: The request must include a valid JWT or API key.
2. **Permission check**: The user must have the required RBAC permission for the operation within the org.
3. **Owner check**: The authenticated user must be the thread owner. Users can only access their own threads.

## Threads Web App

The Threads web app is a browser-based conversation interface designed for end users. Unlike the Admin Dashboard, which is built for platform administrators to manage organizations, sandboxes, and infrastructure, the Threads app provides a streamlined experience focused on sandbox sessions and conversations.

Users sign in via social login (GitHub, GitLab, Google, Vercel), select an organization, and interact with sandbox sessions through the app. The app includes user-level settings for appearance and notification preferences.

The app supports creating and managing threads, viewing message history, and branching conversations. It shares the same authentication and API layer as the rest of the platform.
